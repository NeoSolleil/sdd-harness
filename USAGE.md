# USAGE.md — 運用ガイド（セットアップ後、機能をどう作るか）

セットアップ済みのプロジェクトで、**AI と一緒に機能を1本作る**ときの人間（あなた）向け運転マニュアル。
クローン直後の準備は [SETUP.md](SETUP.md)、構成台帳は [HARNESS.md](HARNESS.md)、規約の正本は [CLAUDE.md](CLAUDE.md)。

> 基本の型: **あなたが指示 → AI が起草 → あなたがレビューして質問に答え、承認（`Status: Approved`）→ 次の段階**。
> AI は各段階の成果物を作りますが、**決定と承認はあなたの仕事**です（ハーネスがそれを強制します）。

---

## 0. 毎回の前提チェック（作業を始める前に）

- [ ] **このリポジトリのフォルダをワークスペースのルートとして開いている**（親フォルダごと開くと `.claude/settings.json` が読まれず、**ガードが一切効かない**）。
- [ ] `pre-commit install` 済み（コミット時強制。SETUP.md 手順5）。
- [ ] （初回のみ）**ガードの実弾確認**: AI に「`backend/app/domain/` に `import fastapi` を書いてみて」と頼み、**ブロックされる**ことを確認。ブロックされなければ § 5 へ。

## 1. 全体の流れ（1機能 = 5段階 + レビュー）

| 段階 | あなたが言うこと（例） | AI が出すもの | あなたがすること |
|---|---|---|---|
| 0. discover | `/discover スコアを記録して一覧で見たい` | `specs/NNNN-name/discovery.md`（Example Map 草案） | **🔴疑問に答える** → 承認 |
| 1. specify | `/specify 0001-score-record` | `requirements.md`（EARS）＋ `acceptance.feature`（Gherkin） | 網羅を確認 → 承認 |
| 2. plan | `/plan 0001-score-record` | `design.md`（DDD・API・データモデル） | **設計の論点に答える** → 承認 |
| 3. tasks | `/tasks 0001-score-record` | `tasks.md`（ビルド順の作業分解） | 順序・網羅を確認 → 承認 |
| 4. implement | `/implement 0001-score-record` | テスト＋実装（TDD）→ レビュー3体 | 待つ → 最終確認・コミット指示 |

前段が `Approved` になるまで次段は**フックが物理的に着手させません**。順番を覚える必要はなく、間違えたら止まります。

## 2. 段階ごとの運転方法

### 段階0: discover（発見）— あなたの関与が最も大きい段階

- **始め方**: `/discover <アイデアや要望を自由文で>`。AI がアナリスト3体（価値・実現性・QA）を並行で走らせ、Example Map（🟡Story / 🔵Rule / 🟢Example / 🔴Question）の草案と `specs/NNNN-kebab-name/` フォルダを作ります。
- **あなたの主な仕事は 🔴Question への回答**です。AI は答えを発明しない規律なので、放置すると先に進みません。「はい/いいえ」か具体値で答えます。
- **承認前に見るポイント**:
  - 価値が1文で言えているか（「誰の・何を・なぜ」）
  - スコープの「含む／含まない」が自分の意図と合っているか（**削れるものは削る**）
  - 🔵 が数件〜十数件か（1件なら小さすぎ＝要件レベル、多すぎ・複数ゴール混在なら機能を分割）
  - 各 🔵 に具体例 🟢 があるか、🔴 が全て解決済みか

### 段階1: specify（仕様化）

- **見るポイント**: 🔵 が全て要件 `R-x` に、🟢 が全てシナリオになっているか（取りこぼし）。曖昧語（「速い」「適切」）が残っていないか。各シナリオに `@R-x` と `@backend`／`@e2e` タグがあるか。
- 承認対象は `requirements.md`（`acceptance.feature` はこれに連動して凍結されます）。
- **以後の仕様の正本はこの2ファイル**です（discovery.md は発見の記録として残るだけ）。

### 段階2: plan（設計）／段階3: tasks（作業分解）

- design.md / tasks.md には **「人間ゲート向けの論点」の節**があります。AI が「ここは判断が要る」と挙げた設計・順序のリスクに答えてください。**あなたの決定は文書に追記され、記録として残ります**。
- 見るポイント: 全 `R-x` が設計要素（plan）・タスク（tasks）に対応しているか（トレーサビリティ表に穴が無いか）。

### 段階4: implement（実装）— あなたは基本「待ち」

- AI が engineer に委譲し、シナリオをテスト化（red）→ 最小実装（green）→ 整理（refactor）で進めます。CA 違反・直接 SQL・仕様外機能はフックとゲートが止めます。
- 実装が緑になると**独立レビュアー3体**（仕様適合・テスト網羅・コード品質）が自動で回り、指摘→修正→再レビューを **3体 PASS まで**繰り返します。
- あなたは最後に結果（テスト・ゲート・レビュー報告）を確認し、コミットを指示します。

## 3. 承認と差し戻しの操作

**承認** = 文書先頭の `Status: Draft` を `Approved` に変えること。方法はどちらでも:

- 自分でエディタで書き換える（これが「人間の承認」の素直な形）
- AI に「承認します。Approved にして」と指示する → **確認ダイアログ（ask）が出るので「許可」**（AI の自己承認を防ぐ人間ゲート。あなたの指示によるものだけが通る仕組みです）

**差し戻し**（承認済みの仕様を変えたくなったら）:

```
1. 対象文書の Status: Approved → Draft に戻す   ← Status 行の編集だけはゲートが許可
2. 内容を修正（AI に指示してよい）
3. レビューして再び Approved に
※ 下流の文書（design/tasks 等）に影響するなら、下流も同様に差し戻して伝播させる
```

## 4. ガードとの付き合い方（ブロックやダイアログが出たら）

| 出るもの | 場面 | 意味・どうするか |
|---|---|---|
| **DENY**（操作がブロックされる） | CA 違反 import・直接 SQL・前段未承認での着手・承認済み文書の編集・`--no-verify` 等 | 正常動作。**エラーメッセージが正しい経路を教えます**（例: 差し戻しフローへ、profile 側を編集へ） |
| **ask**（確認ダイアログ） | `Status: Approved` の書き込み／ハーネス強制層（hooks・settings・profile.yml・CI 等）の変更 | **あなたが指示した操作なら「許可」**。心当たりが無ければ「拒否」して AI に理由を聞く |
| pre-commit の失敗 | コミット時に lint・型・アーキ検証・テスト・ガード自己テストが不合格 | 迂回せず（`--no-verify` はどのみちブロック）、AI に失敗ログを見せて直させる |

## 5. トラブルシューティング

- **ガードが一切反応しない（違反が素通りする）**
  1. ワークスペースを**このリポジトリのルート**で開いているか（親フォルダで開くのが最頻の原因）
  2. `python` が実行できるか（Windows は `python3` が Store のスタブで動かないことがある。フック登録は `python … || python3 …` の両対応だが、両方とも実行不能だと不発）
  3. `.claude/settings.json` を変えた直後はセッション（エディタ）を再起動したか
- **承認済み文書を編集しようとして拒否される** → 仕様変更は § 3 の差し戻しフロー経由が唯一の道（これが正常）。
- **pytest が「テスト0件」で終わる／CI が exit 5 を許容している** → 最初の機能が入るまでの意図された許容です。
- **ハーネス自体を直したい**（ガード・プロファイル・スキル） → [HARNESS.md](HARNESS.md) の「修正の適用パス」（どこを直すと・いつ効いて・何が検証するか）を参照。`.claude/rules/` 等の生成物は直接編集しても上書きされます（profile 側を直す）。

## 6. 2本目以降の機能・並行開発

- 新しい機能はまた `/discover` から。フォルダは AI が既存の最大番号＋1 で採番します（`0002-…`）。
- **ゲートは機能フォルダ単位**なので、別機能どうしの並行（0001 は implement 中、0002 は discover 中）は問題ありません。
- 機能の切り方に迷ったら: そこから EARS 要件が数件〜十数件出る大きさが1機能の目安（1件しか出ないなら要件、複数ゴール混在なら分割）。
