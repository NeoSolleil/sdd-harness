SDD Harness — Call Map

どのエージェントが、どのスキルを呼ぶのか

段階駆動スキルは進行役だけが起動する「台本」。ワーカーは参照スキルだけを呼ぶ。逆はしない。 段階駆動スキル（discover〜implement）は disable-model-invocation: true の slash 専用 — サブエージェントからは技術的に呼べない。craft の知識は参照スキルとして各ワーカーに preload される。

凡例登場する4つの役割

進行役（メインループ）— slash で台本を読み、ワーカーを spawn する ワーカー役（6体）— 委譲で起動され、成果物を作る レビュー役（3体）— implement 後に独立検証する 人間ゲート — 各段階の承認（Status: Approved）

01全体フロー — 5段階の実行順序

各段階は「人間が /コマンド → 進行役が台本（段階駆動スキル）を読む → ワーカーを Agent ツールで spawn → 成果物 → 人間承認」で1サイクル。前段が Approved になるまで次段はフックが物理的にブロックする。

STAGE 0 discover — 発見 /discover <アイデア> 成果物: specs/NNNN-name/discovery.md（Example Map）
進行役が読む台本 skills/discover Example Mapping の進行・Ready 基準・委譲指示
spawn するワーカー（3体並行・1メッセージ）
product-analyst quality-analyst solution-analyst
参照: ubiquitous-language bdd（quality のみ）
各自の観点（🟡🔵🟢🔴）をテキストで返すだけ。ファイルは書かない。
統合（唯一の例外） 進行役が統合 3観点を discovery.md に1箇所で記述（ファイル競合の回避）。🔴疑問は人間に提示。
🚧 人間承認 discovery.md → Status: Approved 🔴疑問がゼロになるまで specify に進めない
STAGE 1 specify — 仕様化 /specify <feature> 成果物: requirements.md（EARS）＋ acceptance.feature（Gherkin）
進行役が読む台本 skills/specify 3段構造・トレーサビリティ・差し戻し規則
spawn するワーカー
scenario-author
参照: bdd ubiquitous-language specify/ears.md（Read）
🔵Rule→EARS 要件、🟢Example→Gherkin シナリオへ清書。発明禁止（不足は discover へ差し戻し）。
出力 requirements.md acceptance.feature Gherkin の一次著作はここ1回だけ。以後は参照・変換のみ。
🚧 人間承認 requirements.md → Status: Approved acceptance.feature は連動して凍結
STAGE 2 plan — 設計 /plan <feature> 成果物: design.md（DDD・API 契約・データモデル）
進行役が読む台本 skills/plan
spawn するワーカー
solution-analyst
参照: ubiquitous-language backend-architecture P frontend-architecture P design P
stack 設計スキルは Skill ツールで随時呼ぶ（P = active プロファイル提供）。
出力 design.md 人間ゲート向けの論点を明示。決定は文書に追記。
🚧 人間承認 design.md → Status: Approved 設計の論点に人間が回答してから承認
STAGE 3 tasks — 作業分解 /tasks <feature> 成果物: tasks.md（TDD ビルド順の作業リスト）
進行役が読む台本 skills/tasks
spawn するワーカー
solution-analyst
参照: ubiquitous-language
既存シナリオを @R-x で参照・グルーピングするだけ。新規シナリオ・要件は作らない。
出力 tasks.md 内側の層→外側の順・test-first で順序付け。
🚧 人間承認 tasks.md → Status: Approved 承認されるまで implement のフックが着手をブロック
STAGE 4 implement — 実装（TDD） /implement <feature> 成果物: テスト＋最小実装（red→green→refactor）
進行役が読む台本 skills/implement 委譲・完了後レビューのループを駆動
spawn するワーカー（独立タスクなら並行）
backend-engineer P frontend-engineer P
参照: backend-architecture P frontend-architecture P e2e-testing P bdd ubiquitous-language design P
委譲プロンプトで担当タスク（tasks.md の ID）と対象シナリオ（@R-x）を受け取る。
完了後レビュー（3体並行）
spec-compliance test-coverage code-reviewer
詳細は下の「03 implement のループ」へ。
🏁 3体 PASS ＋ 人間の最終確認 → コミット指示

02マトリクス — 誰がどのスキルを呼ぶか（一目で規約が見える表）

左の藍色ブロック（段階駆動）に印が付くのは進行役の行だけ、右の緑ブロック（参照）に印が付くのはワーカー／レビュー役の行だけ。 この「対角の空白」が呼び出し規約そのもの。

03implement のループ — 実装〜レビューの反復

実装が緑になるたびにレビュー3体が並行で回り、指摘があれば engineer に再委譲。3体すべて PASS ＋人間承認まで反復する。

進行役 ① engineer へ委譲 tasks.md の ID ＋ 対象シナリオ @R-x を明記して spawn
→
engineer ② red → green → refactor シナリオを選択的にバインド → 失敗確認 → 最小実装 → 品質ゲート緑
→
レビュー役 ×3（並行） ③ 独立検証 spec-compliance ／ test-coverage ／ code-reviewer。確信度の低い指摘は general-purpose を1段だけ spawn して反証
→
人間ゲート ④ 3体 PASS → 最終確認 テスト・ゲート・レビュー報告を確認してコミット指示
指摘あり → 該当 engineer に再委譲して修正 → レビュー再実行（③へ戻る）
補足（規約の細部）
  • 委譲は文章上の約束ではなく、進行役が Agent ツールで実際に spawn する。サブエージェントは会話履歴を引き継がないため、委譲プロンプトに feature フォルダ・入力・期待出力を明記する。独立した複数の委譲は1メッセージで並行に。
  • 進行役は段階成果物を自分で書かない。唯一の例外は discover の統合（3アナリストのテキストを discovery.md に1箇所でまとめる）。
  • P 付きは stack profile（②）が正で、apply.py が .claude/ へ配置する生成物（④）。直すときは必ず profiles/<id>/ 側を編集する。
  • 順序・凍結は散文でなくフックが強制する: 前段が Status: Approved でないと次段の書き込みは guard_sdd_gates が deny。Status: Approved の書き込み自体は人間の確認ダイアログ（ask）。
