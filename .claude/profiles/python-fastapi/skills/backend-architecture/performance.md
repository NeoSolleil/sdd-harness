# performance — 負荷の見積り・負荷テスト・シーディングの手法

「負荷テストの考え方」（SKILL.md）を実際に計画へ落とすときの手法テンプレート。
**数値はすべて記入例**。プロジェクトの想定値に差し替えて design.md か specs/ 側に記録する。

## 1. キャパシティ計画 — Base DAU を唯一の変数にする

想定負荷は **Base DAU（1日あたりアクティブユーザー数）1つを起点**に、係数で機械的に導出する。
数字の根拠が1箇所に集まるので、想定が変わっても表を引き直すだけで済む。

### A. 前提（変数）

| 変数 | 値（例） | 意味 |
| --- | --- | --- |
| **Base DAU** | 50 | 1日のアクティブユーザー数（唯一の入力） |
| Creator Rate | 10% | 新規データ作成（Create）を行うユーザーの割合 |
| Creates / User | 2.0 | 作成ユーザー1人あたりの平均作成数／日 |
| Updates / Entity | 20 | 1エンティティに対する平均更新回数 |

### B. 導出メトリクス（日次）

| メトリクス | 式 | 値（DAU=50） |
| --- | --- | --- |
| Active Creators | `DAU × Creator Rate` | 5 |
| 新規エンティティ総数 | `Active Creators × Creates/User` | 10 /日 |
| 更新総数 | `新規総数 × Updates/Entity` | 200 /日 |

### C. ピークシナリオ（全員が同時に来る瞬間）

| メトリクス | 式 | 値（DAU=50） |
| --- | --- | --- |
| Max CCU（同時接続） | `DAU × 100%` | 50 |
| ピーク書き込み | `Max CCU × 1 req/sec` | 50 req/sec |
| ピーク読み取り | `Max CCU × 2 req/sec` | 100 req/sec |

## 2. 負荷テストシナリオ（locust / k6）

| シナリオ | 内容 | 目的 |
| --- | --- | --- |
| **Smoke** | 1 VU・短時間 | エラー率 0% とベースラインレイテンシの確認 |
| **Spike（最重要）** | 0 → Max CCU（10秒で急増）→ 維持 → 0 | DB コネクションプール枯渇・急増追従性の確認 |
| **Soak** | Max CCU × 20% を1時間 | メモリリーク・長時間稼働でのレイテンシ悪化の検知 |

- 観点は SKILL.md のとおり: スループット（req/s）・p95 レイテンシ・DB クエリ数（N+1）。
- CI 常時実行はしない。設計変更・リリース前など必要時に手動で回す。

## 3. シーディング（試験データの準備）

データ量は**想定運用期間**から逆算する。「空の DB で速い」は当てにならない。

| 変数 | 値（例） | 意味 |
| --- | --- | --- |
| 運用年数 | 3 | 何年運用した状態を再現するか |
| 日数 | 1,095 | `年数 × 365` |

| データ種別 | 式 | 値（例） | 役割 |
| --- | --- | --- | --- |
| **Cold**（過去データ） | `新規総数/日 × 日数` | 10,950 | ノイズ。インデックス性能の検証用 |
| **Hot**（更新対象） | `新規総数/日 × 0.5` | 5 | 負荷試験で実際に読み書きする対象 |

- 関連データ（FK で繋がる行）は整合した状態で同時に投入する。
- 投入は bulk insert（SQLAlchemy なら `insert()` の executemany / `bulk_insert_mappings`）で高速に。
